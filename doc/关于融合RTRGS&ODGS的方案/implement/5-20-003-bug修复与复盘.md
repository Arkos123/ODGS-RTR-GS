### 存在的潜在问题

#### 🟡 等距柱状图像的 `images` 子目录约定

`readCamerasFromOpenMVG` 中图片路径为 `os.path.join('images', cam_dict[cam_key])`。如果数据集目录结构不同（例如 ODGS 的实际 `readCamerasFromOpenMVG` 中也是 `os.path.join('images', ...)`），需要用户确保数据集目录符合 OpenMVG 规范。

#### 🟡 `pseudo_normal` 精度

Python 实现的 `_compute_pseudo_normal` 用了 `kornia.spatial_gradient`（Sobel 算子），与 CUDA 硬件的 `computer_pseudo_normal` 实现略有差异。`mse_loss` 引导法线趋于一致，质量差异在可接受范围内。

## 最终复盘

### 当前训练等距柱状全景图的完整流程

```
Step 0: ODGS 预训练
  python train.py -s <panorama_dataset> -m <odgs_output>

Step 1: PLY 转换
  python script/odgs2rtrgs.py -m <odgs_output> -i 30000 -o <output_dir>
  → 输出 <output_dir>/point_cloud.ply

Step 2: Stage1 等距柱状训练
  python train.py -t render_ref_equirect \
      -s <panorama_dataset> -m <output>/stage1 \
      --ply_checkpoint <output_dir>/point_cloud.ply \
      --ref_map --compute_with_prt \
      --diffuse_iteration 3000 --skip_eval

Step 3: 遮挡烘培
  python baking.py --checkpoint <output>/stage1/checkpoint/chkpnt30000.pth

Step 4: Stage2 PBR 等距柱状训练
  python train.py -t render_ref_pbr_equirect \
      -s <panorama_dataset> -m <output>/stage2 \
      -c <output>/stage1/checkpoint/chkpnt30000.pth \
      --occlusion_path <output>/stage1/checkpoint/occlusion_volumes.pth \
      --ref_map --compute_with_prt --iterations 40000
```

### 核心架构决策：不改 CUDA

融合 ODGS + RTR-GS 选择 **不改 CUDA、纯 Python 多通道前向渲染** 方案，理由：

| 方案                                         | 优点                 | 缺点                        |
| ------------------------------------------ | ------------------ | ------------------------- |
| **改 CUDA**（在 ODGS rasterizer 加 feature 支持） | 单 pass 性能好         | 需维护两套 CUDA；开发成本大；与上游脱节    |
| **多通道前向渲染** ✅                              | 零 CUDA 修改；职责清晰；可维护 | 5-pass 约 2.4 倍开销（\~9.5ms） |

在 `densify_until_iter=0`（冻结几何）的设定下，光栅化不是瓶颈。

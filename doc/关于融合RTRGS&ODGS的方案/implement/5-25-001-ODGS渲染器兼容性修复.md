feat: fix ODGS rasterizer compatibility — bridge layer & compilation fixes

修复 equirect 渲染器集成 ODGS CUDA rasterizer 后的兼容性问题，使全景
训练流水线（Stage 1）能够正常运行。

## 核心设计：Rasterizer 桥接层

对于 ODGS rasterizer 与 RTR-GS rasterizer 的返回值形状差异，不采用
"到处打补丁"的方式，而是在 [render_equirect.py](/gaussian_renderer/render_equirect.py#L32-L44)
的 `_run_odgs_rasterizer` 函数中做统一桥接：

```python
def _run_odgs_rasterizer(means3D, means2D, colors_precomp, opacities, scales, rotations,
                         rasterizer, shs=None, cov3D_precomp=None):
    rendered_image, depth, acc, radii, psi, lat, lon = rasterizer(...)
    # bridge: ODGS returns [H, W], convert to [1, H, W] like RTR-GS
    return rendered_image, depth.unsqueeze(0), acc.unsqueeze(0), radii, psi, lat, lon
```

| 返回值 | RTR-GS rasterizer | ODGS rasterizer (raw) | 桥接后 |
|--------|------------------|----------------------|--------|
| `rendered_image` | `[3, H, W]` | `[3, H, W]` ✅ | `[3, H, W]` |
| `depth` | `[1, H, W]` | `[H, W]` ⚠️ | `[1, H, W]` ✅ |
| `acc` (opacity) | `[1, H, W]` | `[H, W]` ⚠️ | `[1, H, W]` ✅ |

下游 `render_view` 和 `calculate_loss` 全部使用标准 RTR-GS 形状写法，
无需感知 ODGS 的差异。后续新增代码也不需要额外的形状适配。

## 修改的文件

### 1. `gaussian_renderer/render_equirect.py` — 3 项修复

**① scales 维度：`[N,1]` → `[N,3]`**

- 问题：ODGS CUDA kernel `forward.cu:160` 声明 `scales` 为 `glm::vec3*`，
  每元素 3 个 float。传入 `[N,1]` 时越界读取 → illegal memory access。
- 修复：`scales.mean(dim=-1, keepdim=True).repeat(1, 3)` 扩展为 `[N,3]`。
- 说明：ODGS 存储 `[N,3]` 各向同性缩放（3 个值相同），取均值后显式 repeat
  保证各向同性。

**② `_run_odgs_rasterizer` 桥接 depth/acc 形状**

- 在函数内部对 rasterizer 返回结果做 `depth.unsqueeze(0), acc.unsqueeze(0)`
- 下游代码无需任何形状适配

**③ `_run_odgs_rasterizer` 去除空 tensor 转换**

- 问题：原代码将 `cov3D_precomp=None` 转成 `torch.Tensor([]).cuda()`，
  ODGS rasterizer 的 Python 检查认为同时传了 scale/rotation 和 cov3D →
  报错 "exactly one"。
- 修复：直接传 `None` 给 rasterizer，由其内部逻辑行为：将 `None` 转为
  `torch.Tensor([])`（CPU 空 tensor），C++ 层 `data_ptr() == 0` 被识别为空指针，
  正确走 scale/rotation 分支。

### 2. `pbr/renderutils/ops.py` — CUDA stubs 链接路径

- 问题：编译 `renderutils_plugin` 时链接器报 `cannot find -lcuda`。
  `libcuda.so` 位于 CUDA 的 `lib64/stubs/` 目录，不在默认搜索路径。
- 修复：从 `CUDA_HOME` 环境变量读取路径，自动追加 `-L$CUDA_HOME/lib64/stubs`。
- 可移植性：自动检测路径是否存在，不依赖硬编码路径。

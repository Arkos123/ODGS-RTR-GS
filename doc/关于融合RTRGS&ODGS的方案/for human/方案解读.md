## ODGS ↔ RTR-GS 融合方案

### 为什么要融合？

ODGS 扩展了 3DGS 使其支持 **等距柱状投影（equirectangular）的全景图（360°）**，而 RTR-GS 是一个 **逆渲染框架**（BRDF/光照分解 + 重光照）。融合后，RTR-GS 将获得 **直接用全景图序列训练** 的能力，这对于室内扫描、VR/AR 等场景非常有用。

---

### 核心可行性：渲染管线的"三层分离"

RTR-GS 的渲染管线有 **清晰的3层架构**，这是融合可行的核心原因：

```
[第1层] Python: 逐高斯颜色计算（**投影无关**）
    - PRT 颜色（漫反射 + 镜面反射）
    - 反射颜色（forward shading）
    - 组装逐高斯属性
          │
          ▼
[第2层] CUDA: 溅射 + Alpha 混合（**投影相关**）
    - 基于瓦片的排序 & 剔除
    - colors_precomp → rendered_image
    - features 张量  → rendered_feature  ← 关键差异点
          │
          ▼
[第3层] Python: 逐像素延迟着色（**投影无关**）
    - 法线图提取
    - 反射渲染（get_reflectance_color）
    - PBR 着色（pbr_shading）
```

**关键洞察**：只有第2层（CUDA 光栅化器）知道投影模型。第1层和第3层完全与投影无关。因此只需要 **替换第2层的 CUDA 光栅化器**（从 `diff_gaussian_rasterization` 换为 `odgs_gaussian_rasterization`），即可实现全景图支持。

---

### 最大的挑战：Feature 张量机制

RTR-GS 的延迟渲染（deferred shading）依赖于 `features` 张量——它将每个高斯的深度、法线、BRDF 参数打包成多通道向量，CUDA 内核对其每个通道独立进行 Alpha 混合，然后逐像素拆分使用。

**ODGS 的 CUDA 光栅化器不支持 `features` 机制**，它只处理 `colors_precomp`（RGB）和 `sh`（球谐系数）。

#### 解决方案：多通道前向渲染

不修改 ODGS CUDA 内核，而是将每个需要的通道作为独立的前向渲染通道：

| 通道 | colors_precomp = | 输出 | 用途 |
|------|-----------------|------|------|
| 1 | PRT 辐射度 (RGB) | `radiance_map` | 基础颜色 |
| 2 | 法线 (编码为RGB) | `normal_map` | 逐像素法线 |
| 3 | BRDF参数 (编码为RGB) | `brdf_map` | base_color, roughness, metallic |
| 4 | 深度 (float→RGB) | `depth_map` | 逐像素深度 |

所有通道使用 **同一个** ODGS 光栅化器、**同一组** 相机参数、**相同** 的高斯数据。渲染后解码回语义属性，运行标准的 RTR-GS 逐像素 PBR 着色。

性能代价：ODGS 光栅化器在 1024×512 下 ~2ms/通道，4 个通道约 8ms。

### 分阶段实施计划

#### 阶段1：PLY 桥接（低难度，~80行）

创建 `scripts/odgs2rtrgs.py`：
1. 加载 ODGS 训练的 `.ply`（6个属性：xyz, SH, scale, rot, opacity）
2. 创建 RTR-GS 的 GaussianModel
3. 将 RTR-GS 扩展属性初始化为默认值：
   - `ref_tint` = 0, `ref_strength` = sigmoid⁻¹(0.01), `ref_roughness` = sigmoid⁻¹(0.65)
   - `diffuse_transfer_dc/rest` = 0, `specular_feature` = 0
   - PBR: `base_color`, `roughness`, `metallic` = 默认值
4. 保存为 RTR-GS 格式的 `.ply`

同时修复 RTR-GS 中 `load_ply` 的变量名 bug。

#### 阶段2：等距柱状相机支持（低难度，~100行）

1. 在 RTR-GS 的 `scene/dataset_readers.py` 中添加全景图数据集读取器（复用 ODGS 的 OpenMVG 解析器）
2. ODGS 的相机和 RTR-GS 的 `Camera(R, T, FoVx, FoVy, ...)` 接口兼容，`FoVx = π` 硬编码

#### 阶段3：等距柱状渲染器（核心工作，中等难度，~300行）

创建 `gaussian_renderer/render_equirect.py`：
1. 复用 RTR-GS 的 PRT 计算（投影无关）
2. 替换 CUDA 光栅化器为 `odgs_gaussian_rasterization`
3. 移除透视特有参数：`tanfovx/tanfovy/cx/cy/projmatrix`
4. 实现 **多通道前向渲染**（替代 `features` 机制）
5. 添加 **纬度加权损失函数**（WS-PSNR/WS-SSIM，`cos(纬度)` 加权）

#### 阶段4：训练流水线改造（中等难度，~100行）

修改 `train.py`：
1. 新增 `--equirect` 模式开关
2. 加载 ODGS `.ply` 或 checkpoint 作为初始几何
3. 冻结密化（类似 Stage2 行为）
4. 使用等距柱状渲染器 + 纬度加权损失

```
                    ┌──────────────────────────────────────┐
                    │  ODGS 训练（等距柱状）                │
 输入：全景图序列   │  - 重建 3DGS 几何                     │
 ─────────────────► │  - 输出 .ply + checkpoint             │
                    └──────────┬───────────────────────────┘
                               │
                               ▼
                    ┌──────────────────────────────────────┐
                    │  PLY 转换 +                          │
                    │  RTR-GS 属性初始化                   │
                    └──────────┬───────────────────────────┘
                               │
                               ▼
                    ┌──────────────────────────────────────┐
                    │  RTR-GS 等距柱状逆渲染               │
                    │  - 使用 ODGS CUDA 光栅化器           │
 输入：全景图序列   │  - 多通道前向渲染                    │
 ─────────────────► │  - PRT + 反射 + PBR                  │
                    │  - 纬度加权损失                      │
                    └──────────────────────────────────────┘
```

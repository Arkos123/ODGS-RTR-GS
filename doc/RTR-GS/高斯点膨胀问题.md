
好的，我已经找到了根本原因。这是一个**正常传播阶段的 scale 放大 + 激进剪枝**导致的连锁反应。

## 问题分析

从终端输出可以看到两条关键证据：

| 时刻 | 高斯点数 | PSNR |
|---|---|---|
| iter 3998 | **182,637** | 18.76 |
| iter 5289 | **40,410** | 11.71 |

在 ~1300 步内，**78% 的高斯点被剪掉了**，PSNR 骤降 7 dB。

### 时间线还原

根据 [train.py](file:///home/huangpengyue/projects/RTR-GS/train.py#L217-L243) 的训练调度逻辑：

```
iter 3000: reset_opacity() — 透明度钳制到 0.01（正常）
iter 4000: reset_opacity1() + reset_scale() ← 🔥 问题触发点
iter 4100: densify_and_prune → 尺度放大后的高斯被大量剪除
iter 4200~4900: 持续剪枝...
iter 5000: reset_opacity1() + reset_scale() ← 🔥 第二次放大
iter 5100~5300: 持续剪枝... 最终只剩 40k
```

### 根因：`reset_scale()` + 剪枝条件的冲突

在 [gaussian_model.py:L660-L674](file:///home/huangpengyue/projects/RTR-GS/scene/gaussian_model.py#L660-L674)，`reset_scale` 会把非反射高斯的 **2 个非最小轴放大 1.5 倍**：

```python
def enlarge_refl_scales(self, ENLARGE_SCALE=1.5, ...):
    min_axis_id = torch.argmin(scales, dim=-1, keepdim=True)
    rmin_axis = (torch.ones_like(scales)*ENLARGE_SCALE).scatter(1, min_axis_id, 1)
    scale_new = self.scaling_inverse_activation(scales*rmin_axis)
```

放大后的高斯在屏幕空间变大，然后在 [gaussian_model.py:L1089-L1108](file:///home/huangpengyue/projects/RTR-GS/scene/gaussian_model.py#L1089-L1108) 的剪枝中触发两个条件：

1. **`max_radii2D > 20`** — iter 3000 后启用（`size_threshold=20`）
2. **`scaling.max() > 0.1 * cameras_extent`** — 对于 360Roam（extent≈3.28），即 scaling > 0.328

### 为什么 360Roam 比其他场景更严重？

360Roam 只有 **4 个 cube 面**（F/B/L/R），视角覆盖远不如 MipNerf360 的 100+ 张多角度图片。视角少 → 高斯约束弱 → scale 放大后更难被梯度纠正回来 → 更容易触发剪枝阈值。

## 修复方案

有三个未注释的选项，我推荐**方案 1**（最简单、最安全）：

### 方案 1（推荐）：禁用 scale 放大 — `--without_normal_propagation`

在 Stage 1 命令中添加 `--without_normal_propagation`，这会跳过 iter 4000/5000/... 的 `reset_scale()` 和 `reset_refl()` 调用，只保留 iter 3000/6000/... 的标准 `reset_opacity()`。

**代价**：法线传播阶段的尺度扩大被跳过，可能略微影响法线质量。但对于只有 4 个 cube 面的场景，这个代价远小于高斯坍缩的风险。

### 方案 2：降低 ENLARGE_SCALE 因子

修改 [gaussian_model.py:L660](file:///home/huangpengyue/projects/RTR-GS/scene/gaussian_model.py#L660) 的默认值，从 `ENLARGE_SCALE=1.5` 降到 `1.1` 或 `1.2`：

```python
def enlarge_refl_scales(self, ret_raw=True, ENLARGE_SCALE=1.2, ...):
```

这会让放大更温和，不至于触发剪枝阈值。

### 方案 3：只在特定场景禁用

不改代码，只修改 `run_360roam.sh`，后续如果其他场景需要 scale 放大可以保留。
